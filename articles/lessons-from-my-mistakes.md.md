## 🌱 機能がてんこ盛りで実装が間に合わない

開発者も企画段階に参画して技術的問題の確認を行います。
実現性が低い場合は、仕様の見直しやリリース日の調整を行う

| 項目 | 内容 |
|------|------|
| 何が起きたか | 実現性を確認せず、開発途中で実現性が低いことが発覚した |
| 本当の原因 | 技術者を呼ばず、企画を進めてしまった |
| 次からどうすれば | 企画段階で技術者を呼び実現性について議論する |

:::message
**私の体験談**
お客さんから「○○データも一緒に表示したい」という要望に対して安易に"可能"と回答してしまい、
いざ開発を始めるとデータベースのどこにもその値がありませんでした。

**✍追記**
追加要望を無償で対応するような雰囲気だったため、追加要望に対して適切に請求していれば、ユーザーの要望も減ったかもしれません。
:::

## 🌱 お願いされた機能を断れない
| 項目 | 内容 |
|------|------|
| 何が起きたか | 追加要望を随時仕様に追加し、スケジュールの遅れを招いた。 |
| 本当の原因 | スケジュールのバッファーに追加要望を対応する計画をしていた |
| 次からどうすれば | 追加要望が発生した場合は、ステークホルダーと相談してタスクの優先順位を確認する |

:::message
**私の体験談**
お客さんと一緒に要望や改善点を出して日々より良いサービスに育てましょう！と営業とリーダーがお客さんに口約束をしました。
結果的に、「あの要望はいつになったらできますか？」と催促の連絡が後を絶ちませんでした。

:::

## 🌱 開発のことしか書かれていないスケジュール
| 項目 | 内容 |
|------|------|
| 何が起きたか | リリース日直前に開発以外のタスクが発覚した |
| 本当の原因 | 計画時に開発以外のタスクをスケジュールに含めていない |
| 次からどうすれば | 機能に対してWBSを行い開発以外のタスクを洗い出す |

:::message
**私の体験談**
開発そのものはスケジュール通りに終わらせることができましたが、
テーブルのカラム追加の確認タスクを失念し、慌てて実行するSQLの確認や影響度チェックをリリース前にした経験があります。

**✍追記**
今ならAIと対話することができるので、AIにサポートしてもらえば抜け漏れが昔よりも少なくなるかもしれません。
:::

## 🌱 教育コストをスケジュールに組み込んでいない

新人をサポートするベテランの教育コストをスケジュールに組み込めず、ベテランのタスクが遅延しました。

| 項目 | 内容 |
|------|------|
| 何が起きたか | 新人の開発能力を考慮してスケジュールを立てましたが、遅延しました。 |
| 本当の原因 | 新人をサポートするベテランの教育コストをスケジュールに組み込んでいなかった |
| 次からどうすれば | ベテラン側の教育コストもスケジュールに組み込み |

:::message
**私の体験談**
4月に入社した3人の新人の教育係を担当することになりましたが、
一日の大半が教育とレビューで終わり、自分のタスクに着手できませんでした。
:::

## 🌱 一般的な技術なら課題がないと錯覚する。
| 項目 | 内容 |
|------|------|
| 何が起きたか | 世間一般的な技術を楽観的に捉えて調査や技術検証をないがしろにした |

結果、欠陥が多かったです。
:::
