## 🌱 ドメイン駆動設計のメリット
まず、ドメイン駆動設計をシステム開発に導入すると
どんな良い事が発生するのかを整理します。

### 1. 限られたリソースを重要なタスクに専念できる
ドメイン駆動設計の分析手法として`業務領域のカテゴリー分け`があります。
簡単に説明すると、`3つの領域`に分類して開発方針を決める手段です

:::message
- **中核の業務領域**: そのビジネスの**ストロングポイント**の要素となり、他サービスを差別化を測る領域
- **一般的な業務領域**: そのビジネスを**成り立たせるために必須な要素**となり、システムの基盤となる領域
- **補完的な業務領域**: 中核領域の付加価値を最大化させる補助的な要素となり、ユーザー体験を補強する領域
:::

つまり、`業務領域のカテゴリー分け`が出来れいれば、
チームで業務をよく理解して人が**中核の業務領域**に専念でき、
**中核の業務領域**以外は、外注や経験が浅いメンバーが対応することが可能です。

### 2. 誤訳が少なくなる
言語として同じ意味だが、**立場によって大事にする意味が変わる**
例えば、「野菜」なら🧑‍🌾農家、🧑‍🍳料理人、🧑‍🔧運送関係者で下記のように考える。

- 農家🧑‍🌾: [ 品種, 播種日, 収穫予定日, 農薬, 収穫量 ]
- 料理人🧑‍🍳: [ 鮮度, 鮮度, 味, 原価, アレルギー情報 ]
- 運送関係者🧑‍🔧: [ 重量, 体積, 設定温度（冷蔵/常温）, 積み上げ厳禁 ]

このまま「野菜」というドメインを作成したら、**神クラスのドメイン**に昇華するため、
今回のケースなら、下記のように作成する

- 農家🧑‍🌾: **農作物** というドメイン
- 料理人🧑‍🍳: **荷物** というドメイン
- 運送関係者🧑‍🔧: **食材** というドメイン

## 🌱 実際に開発して良かったこと
### 1. 考慮漏れを減らせる
値オブジェクトを開発する時に、
**設計**　⇔　**ドキュメント作成**　⇔　**値オブジェクト作成**　を何度も反芻するため、
値の条件や制約を見つめなおします。

例えば、時刻を表す`タイムスタンプ`を簡単に実装するとこんな感じ
```ts
const timestamp: Date = new Date("2026-03-19T10:00:00");
```

しかし、このままでは現実では不可能なことが可能なことができます。
それは、`タイムスタンプ`を**過去の時間にする**や**未来の時間にする**

```ts
// 過去の時間にする
const timestamp: Date = new Date("2025-03-19T10:00:00");

// 未来の時間にする
const timestamp: Date = new Date("2030-03-19T10:00:00");

```

下記フローのときに「」
- **設計**
- **ドキュメント作成**
- **値オブジェクト作成**　


、⇔を意識する事によって深く考えるきっかけ出来た
