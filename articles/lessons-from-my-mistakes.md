## 🌱 はじめに

今回は、`ソフトウェア開発現場の「失敗」を集めてみた。 42の失敗事例で学ぶチーム開発のうまい進めかた`を読みました。

@[card](https://www.seshop.com/product/detail/26209)

自分の経験と酷似する事例があり、とても面白い内容でした。
その中でもこの事例は今後も忘れないように執筆します。

## 🌱 お願いされた機能を断れない
| 項目 | 内容 |
|------|------|
| 何が起きたか | 追加要望を随時仕様に追加し、スケジュールの遅れを招いた。 |
| 本当の原因 | スケジュールのバッファーに追加要望を対応する計画をしていた |
| 次からどうすれば | 追加要望が発生した場合は、ステークホルダーと相談してタスクの優先順位を確認する |

:::message
**私の体験談**
お客さんと一緒に要望や改善点を出して日々より良いサービスに育てましょう！と営業とリーダーがお客さんに口約束をしました。
結果的に「あの要望はいつになったらできますか？」と催促の連絡が後を絶ちませんでした。

:::

## 🌱 開発のことしか書かれていないスケジュール
| 項目 | 内容 |
|------|------|
| 何が起きたか | リリース日直前に開発以外のタスクが発覚した |
| 本当の原因 | 計画時に開発以外のタスクをスケジュールに含めていない |
| 次からどうすれば | WBSを行い開発以外のタスクを洗い出す |

:::message
**私の体験談**
開発そのものはスケジュール通りに終わらせることができましたが、
テーブルのカラム追加の確認タスクを失念し、
慌てて実行するSQLの確認や影響度チェックをリリース前にした経験があります。

**✍追記**
今ならAIと対話することができるので、
AIにサポートしてもらえば抜け漏れが昔よりも少なくなるかもしれません。
:::

## 🌱 教育コストをスケジュールに組み込んでいない
| 項目 | 内容 |
|------|------|
| 何が起きたか | 新人の開発能力を考慮してスケジュールを立てましたが、遅延した |
| 本当の原因 | 新人をサポートするベテランの教育コストをスケジュールに組み込んでいなかった |
| 次からどうすれば | ベテラン側の教育コストもスケジュールに組み込む |

:::message
**私の体験談**
4月に入社した3人の新人の教育係を担当することになりましたが、
一日の大半が教育とレビューで終わり、自分のタスクに着手できませんでした。
:::

## 🌱 一般的な技術なら課題がないと錯覚する
| 項目 | 内容 |
|------|------|
| 何が起きたか | 世間一般的な技術を楽観的に捉えて調査や技術検証をないがしろにした |
| 本当の原因 | その技術はチームで誰も触ったことがない |
| 次からどうすれば | 誰も触ったことがない技術は新技術と考える |

:::message
**私の体験談**
今はモダンな技術で先進的な会社はみんな使っているという安易な理由で技術選定を行い、
結果として欠陥が多く、苦労しました。
:::

## 🌱 販売されている商品なら問題ないと思い込む


| 項目 | 内容 |
|------|------|
| 何が起きたか | 商品スペックに記載されている最大データ数を投入すると、処理完了にものすごく時間がかかった |
| 本当の原因 | 商品スペックを信用して動作確認を行わなかった |
| 次からどうすれば | 実際に商品を動かしてみて期待どおりかを確認する |

:::message
**私の体験談**
  Webアプリのセキュリティを検査してくれる製品をチームで購入しましたが、
まったくインシデントを検知してくれませんでした。
(プログラムの中身を知っている開発者からするとバックドアだらけのサービスだったのに...)
:::
